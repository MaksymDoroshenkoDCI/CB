## **1. Introduction and Overview**

This document provides a comprehensive technical overview of the Avto-Polis system. It details the system's architecture, technology stack, features, and operational guidelines for developers, administrators, and other technical stakeholders.

### **1.1. System Purpose and Main Goals**

**Avto-Polis** is an online insurance aggregator platform designed to simplify the process of purchasing vehicle insurance. The system enables users to compare insurance products from various providers, calculate premiums, and purchase policies directly through a web-based interface.

The primary goals of the Avto-Polis project are:
*   **Simplify Choice:** To provide a single platform for users to compare multiple insurance offers, eliminating the need to visit different insurance company websites.
*   **Transparency:** To present insurance product details, pricing, and terms in a clear and understandable format.
*   **Convenience:** To offer a fully online, end-to-end process from quote calculation to policy issuance, accessible from any device.
*   **Efficiency:** To automate the process of obtaining insurance, saving time for both customers and insurance partners.

### **1.2. Target Audience**

The system is designed for two primary user groups:

*   **Customers (Car Owners):** Individuals seeking to purchase or renew motor third-party liability insurance ("Avtotsyvilka") and other vehicle-related insurance products. They require a simple, fast, and intuitive user interface.
*   **System Administrators:** Internal staff responsible for managing the platform's content, partners, tariffs, and user-submitted applications. They require a robust administrative back-office to oversee operations.

### **1.3. Key Features Summary**

The Avto-Polis system will provide the following core functionalities:
*   **User Account Management:** Secure registration and personal account area for managing personal data, vehicles, and policies.
*   **Insurance Premium Calculator:** An interactive tool to calculate the cost of motor third-party liability insurance based on vehicle and driver parameters.
*   **Insurance Offer Comparison:** A feature to display and compare offers from multiple integrated insurance companies side-by-side.
*   **Online Policy Purchase:** An end-to-end workflow for selecting an offer, providing necessary data, making a payment, and receiving the electronic policy.
*   **Policy Management:** The ability for users to view their active policies, access policy documents, and receive automated reminders for policy renewal.
*   **Administrative Dashboard:** A secure panel for administrators to manage insurance partners, configure tariffs, and process user orders.

## **2. System Architecture**

### **2.1. Overall Architecture Description**

The Avto-Polis system will be implemented using a **Service-Oriented Architecture (SOA)** with a decoupled frontend and backend. This approach, often referred to as a **Single Page Application (SPA) with a RESTful API**, provides flexibility, scalability, and a clear separation of concerns.

*   The **Frontend** is a client-side application responsible for all user interface rendering and user interactions.
*   The **Backend** is a server-side application that exposes a set of RESTful API endpoints. It handles all business logic, data processing, and communication with the database and third-party services.

### **2.2. Main Components and Modules**

1.  **Frontend Application (SPA):** A modern JavaScript application built with a framework like React or Vue. It communicates with the Backend via REST API calls.
2.  **Backend API Server:** A web server application built with a framework like Django (Python) or Laravel (PHP). It serves as the core of the system.
3.  **Database:** A PostgreSQL relational database to store all system data, including user profiles, vehicles, policies, and orders.
4.  **Third-Party Insurance APIs:** External services provided by partner insurance companies. The backend will integrate with these APIs to fetch quotes and issue policies.
5.  **Authentication Service:** A module within the backend responsible for handling user authentication and authorization using JSON Web Tokens (JWT).

### **2.3. System Diagrams (Text Format)**

#### **High-Level Component Diagram**

```
 [User's Browser] <--> [Frontend Application (React/Vue SPA)]
       ^
       | (HTTPS - REST API Calls)
       v
 [Backend API (Django/Laravel)] <--> [PostgreSQL Database]
       ^
       | (HTTPS - REST API Calls)
       v
 [External Insurance Company APIs]
```

### **2.4. Data Flow**

A typical data flow for calculating an insurance premium:
1.  **User Input:** The user navigates to the calculator page on the Frontend Application and enters vehicle details (e.g., make, model, year, engine size).
2.  **API Request:** The Frontend sends a `POST` request with the vehicle data to the Backend API endpoint (e.g., `/api/v1/calculator/calculate`).
3.  **Backend Processing:**
    *   The Backend receives the request and validates the input data.
    *   It iterates through the list of integrated insurance partners.
    *   For each partner, it makes a separate API call to their external API, sending the vehicle data in the required format.
4.  **External API Response:** Each insurance company's API returns a quote (premium price and coverage details).
5.  **Data Aggregation:** The Backend aggregates the responses from all partners into a standardized JSON format.
6.  **API Response:** The Backend sends a single JSON response back to the Frontend containing a list of comparable insurance offers.
7.  **Display Results:** The Frontend parses the JSON response and displays the offers to the user in a clear, comparable table or list format.

## **4. Functionality**

### **4.1. Detailed Feature Descriptions**

#### **Module 1: User Account & Profile**
*   **User Registration:** Users can sign up using an email and password. Passwords must be securely hashed.
*   **User Login:** Authenticates users using JWT. The system provides an access token and a refresh token.
*   **Personal Cabinet:** A dedicated dashboard for authenticated users.
    *   **My Cars:** Users can add, edit, and delete their vehicle information (VIN, registration number, make, model, etc.).
    *   **My Policies:** Users can view a list of their active and expired policies, with an option to download the policy document.
    *   **Order History:** A history of all completed and pending insurance purchases.

#### **Module 2: Insurance Calculation and Comparison**
*   **Calculator:** A public-facing tool that requires no authentication. Users input vehicle parameters to get an estimated premium.
*   **Offer Comparison:** After calculation, the system displays a list of offers from different insurance companies. Each offer will clearly show the company name, price, and key benefits.

#### **Module 3: Online Purchase Workflow**
*   **Checkout:** A multi-step process where the user provides personal details (name, ID, address) and confirms the chosen insurance offer.
*   **Payment Integration:** Integration with a secure payment gateway (e.g., Stripe, Fondy, LiqPay) to process online payments.
*   **Policy Issuance:** Upon successful payment, the backend makes a final API call to the insurance partner to issue the policy. The electronic policy (PDF) is then generated and made available to the user in their personal cabinet and sent via email.

#### **Module 4: Notifications**
*   **Email Notifications:** The system will send automated emails for:
    *   Successful registration.
    *   Successful order placement.
    *   Policy issuance.
    *   Upcoming policy expiration reminders (e.g., 30, 15, and 3 days before expiry).

#### **Module 5: Admin Panel**
*   **Dashboard:** An overview of key metrics (orders, revenue, new users).
*   **Partner Management:** CRUD (Create, Read, Update, Delete) operations for insurance company partners, including their API credentials and settings.
*   **Tariff Management:** Interface to manage and adjust insurance tariff coefficients if not fully automated via partner APIs.
*   **Order Management:** View and manage all user orders, check their status (pending, paid, completed, failed), and manually assist if needed.

### **4.2. Use Cases**

*   **Use Case 1: New User Purchases a Policy**
    1.  User calculates a premium.
    2.  User compares offers and selects one.
    3.  User is prompted to register or log in.
    4.  User completes the checkout form with personal and vehicle data.
    5.  User is redirected to the payment gateway and completes the payment.
    6.  User receives the policy via email and can access it in their personal cabinet.

*   **Use Case 2: Existing User Renews a Policy**
    1.  User receives an email reminder about an expiring policy.
    2.  User logs into their personal cabinet.
    3.  User navigates to the "My Policies" section and clicks "Renew" on the expiring policy.
    4.  The system pre-fills the calculator with the existing car's data and shows current offers.
    5.  User selects an offer and proceeds directly to payment.
    6.  The new policy is issued and becomes active upon the old one's expiration.

## **5. API and Integrations**

The system's backend will expose a RESTful API. All communication between the frontend and backend will be in JSON format over HTTPS.

### **5.1. Example API Endpoints**

| Method | Endpoint                             | Description                                            | Auth Required |
| ------ | ------------------------------------ | ------------------------------------------------------ | ------------- |
| `POST` | `/api/v1/auth/register`              | Create a new user account.                             | No            |
| `POST` | `/api/v1/auth/token`                 | Obtain a JWT access/refresh token pair.                | No            |
| `POST` | `/api/v1/auth/token/refresh`         | Refresh an expired access token.                       | Yes (Refresh) |
| `POST` | `/api/v1/calculator/calculate`       | Calculate insurance premiums for a given vehicle.      | No            |
| `GET`  | `/api/v1/policies/`                  | Get a list of the authenticated user's policies.       | Yes           |
| `POST` | `/api/v1/orders/`                    | Create a new insurance order.                          | Yes           |
| `GET`  | `/api/v1/orders/{id}/`               | Get details of a specific order.                       | Yes           |
| `POST` | `/api/v1/orders/{id}/payment`        | Initiate the payment process for an order.             | Yes           |

### **5.2. External Service Integrations**

*   **Insurance Company APIs:** This is the most critical integration. The system must be designed with a flexible "adapter" or "connector" pattern to accommodate different API specifications from various insurance partners. Each adapter will be responsible for:
    *   Transforming our internal data model to the partner's API request format.
    *   Handling the specific authentication method of the partner's API.
    *   Parsing the partner's API response and transforming it into our standardized offer format.
*   **Payment Gateway API:** Integration with a payment provider to handle credit card transactions securely.
*   **Email Service API:** Integration with a transactional email service (e.g., SendGrid, Mailgun) for reliable email delivery.

## **6. Database Design**

The database schema will be designed to support the core functionalities of the platform.

### **6.1. Data Models (Simplified)**

*   **`User`**
    *   `id` (PK)
    *   `email` (UNIQUE, VARCHAR)
    *   `password_hash` (VARCHAR)
    *   `first_name` (VARCHAR)
    *   `last_name` (VARCHAR)
    *   `role` (ENUM: 'customer', 'admin')
    *   `created_at`, `updated_at`

*   **`Vehicle`**
    *   `id` (PK)
    *   `user_id` (FK to `User`)
    *   `make` (VARCHAR)
    *   `model` (VARCHAR)
    *   `year` (INTEGER)
    *   `vin` (VARCHAR, UNIQUE)
    *   `registration_number` (VARCHAR)
    *   `created_at`, `updated_at`

*   **`InsuranceCompany`**
    *   `id` (PK)
    *   `name` (VARCHAR)
    *   `api_config` (JSONB) - Stores credentials, endpoints, etc.
    *   `is_active` (BOOLEAN)

*   **`Policy`**
    *   `id` (PK)
    *   `user_id` (FK to `User`)
    *   `vehicle_id` (FK to `Vehicle`)
    *   `company_id` (FK to `InsuranceCompany`)
    *   `order_id` (FK to `Order`)
    *   `policy_number` (VARCHAR)
    *   `start_date` (DATE)
    *   `end_date` (DATE)
    *   `status` (ENUM: 'active', 'expired', 'cancelled')
    *   `premium_amount` (DECIMAL)
    *   `policy_document_url` (VARCHAR)

*   **`Order`**
    *   `id` (PK)
    *   `user_id` (FK to `User`)
    *   `status` (ENUM: 'pending', 'paid', 'completed', 'failed')
    *   `total_price` (DECIMAL)
    *   `created_at`, `updated_at`

### **6.2. Relationships**
*   A `User` can have many `Vehicles`.
*   A `User` can have many `Policies`.
*   A `Policy` belongs to one `User`, one `Vehicle`, and one `InsuranceCompany`.
*   An `Order` is linked to a `User` and results in one `Policy`.

## **7. Security**

### **7.1. Authentication Mechanisms**

*   **JWT (JSON Web Tokens):** The API will be stateless. Authentication is handled by providing a JWT in the `Authorization: Bearer <token>` header of each request to protected endpoints.
*   **Token Lifecycle:**
    *   An `access_token` with a short lifespan (e.g., 15 minutes) is used for API requests.
    *   A `refresh_token` with a longer lifespan (e.g., 30 days) is stored securely on the client (e.g., in an HttpOnly cookie) and used to request a new `access_token` when it expires.

### **7.2. Authorization Rules**

*   **Role-Based Access Control (RBAC):** The system will implement two primary roles: `customer` and `admin`.
    *   **Customer:** Can only access and manage their own data (vehicles, policies, orders).
    *   **Admin:** Can access the admin panel and manage platform-wide data (partners, users, all orders).
*   API endpoints will be protected by permission classes that verify the user's role and ownership of the requested resource.

### **7.3. Data Protection**

*   **Password Hashing:** User passwords will be hashed using a strong, modern algorithm like **Bcrypt**.
*   **HTTPS/TLS:** All communication between the client, server, and external APIs must be encrypted using TLS.
*   **Environment Variables:** Sensitive information such as database credentials, API keys, and secret keys must be stored in environment variables, not in the codebase.
*   **Input Validation:** All user-supplied data will be rigorously validated on the backend to prevent security vulnerabilities like XSS and SQL Injection.

## **8. Deployment and Infrastructure**

### **8.1. Deployment Process**

Deployment will be automated using a CI/CD pipeline. The general flow for deploying a new feature is:
1.  Developer pushes code to a feature branch in the Git repository.
2.  A Pull Request is created to merge into the `develop` or `main` branch.
3.  The CI/CD pipeline automatically runs linters and executes all automated tests.
4.  Upon successful tests and a code review, the PR is merged.
5.  Merging to the `main` branch triggers an automated deployment to the production environment.

### **8.2. Infrastructure Requirements**

*   **Cloud Provider:** A major cloud provider like AWS, Google Cloud, or Azure.
*   **Web Server:** Nginx to act as a reverse proxy, handle SSL termination, and serve static files.
*   **Application Server:** A server to run the backend application (e.g., Gunicorn for Django).
*   **Database Server:** A managed PostgreSQL instance (e.g., AWS RDS) is recommended for scalability and reliability.
*   **Containerization:** The entire application (frontend, backend) will be containerized using **Docker** to ensure consistency across environments.

## **9. Testing**

### **9.1. Testing Strategy**

A multi-layered testing strategy will be adopted to ensure code quality and system reliability, following the principles of the testing pyramid.

### **9.2. Test Types**

*   **Unit Tests:** To test individual functions, methods, and components in isolation. Frameworks like `pytest` (for Django) or `Jest` (for React) will be used.
*   **Integration Tests:** To test the interaction between different parts of the system, such as API endpoints and the database. The Django REST Framework's test client will be heavily utilized.
*   **End-to-End (E2E) Tests:** To simulate real user workflows in a browser. Tools like **Cypress** or **Playwright** will be used to test critical paths like user registration, policy calculation, and purchase.

## **10. Monitoring and Logging**

### **10.1. Monitoring Approach**

*   **Application Performance Monitoring (APM):** Tools like **Sentry**, **Datadog**, or an open-source stack like **Prometheus/Grafana** will be used to monitor API response times, error rates, and database query performance.
*   **Infrastructure Monitoring:** Cloud provider's built-in monitoring tools will be used to track CPU, memory, and disk usage of servers and databases.

### **10.2. Logging Strategy**

*   **Structured Logging:** All log entries will be in a structured format (e.g., JSON) to allow for easy parsing and searching.
*   **Centralized Logging:** Logs from all services will be aggregated into a centralized logging platform like the **ELK Stack** (Elasticsearch, Logstash, Kibana) or **Graylog**.
*   **Log Levels:** Standard log levels (DEBUG, INFO, WARNING, ERROR, CRITICAL) will be used to control verbosity.

## **11. Developer Guide**

### **11.1. Getting Started**

1.  **Prerequisites:** Ensure you have Git, Docker, and Docker Compose installed on your local machine.
2.  **Clone Repository:**
    ```bash
    git clone https://github.com/your-org/avto-polis.git
    cd avto-polis
    ```
3.  **Environment Variables:** Copy the example environment file and populate it with your local configuration.
    ```bash
    cp backend/.env.example backend/.env
    cp frontend/.env.example frontend/.env
    ```
4.  **Launch Application:**
    ```bash
    docker-compose up --build
    ```
    *   The backend API will be available at `http://localhost:8000`.
    *   The frontend application will be available at `http://localhost:3000`.

### **11.2. Contribution Guidelines**

*   **Branching Strategy:** Use a GitFlow-like model.
    *   `main`: Contains production-ready code.
    *   `develop`: Integration branch for new features.
    *   `feature/your-feature-name`: Create new branches from `develop` for your work.
*   **Pull Requests (PRs):** All code must be submitted via PRs to the `develop` branch. PRs must pass all automated checks and receive at least one approval from another team member before being merged.
*   **Code Style:** Adhere to the established code style guides (e.g., **PEP 8** for Python, **Prettier** for JavaScript/React). A linter will be part of the CI pipeline to enforce this.
