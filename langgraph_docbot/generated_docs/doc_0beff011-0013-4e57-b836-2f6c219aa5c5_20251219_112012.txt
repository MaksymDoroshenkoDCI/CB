### **1. Вступ та загальний опис системи**

#### **1.1. Мета документа**

Цей документ слугує центральним джерелом технічної інформації про сервіс `fintom8`. Його мета — надати вичерпне розуміння архітектури, функціональності, процесів розгортання та підтримки системи для ефективного онбордингу та подальшого розвитку.

**Цільова аудиторія:**
*   **Розробники:** для розробки нових функцій та виправлення помилок.
*   **QA-інженери:** для розуміння логіки роботи та розробки тестових сценаріїв.
*   **DevOps-інженери:** для налаштування інфраструктури, CI/CD та моніторингу.
*   **Архітектори:** для аналізу архітектурних рішень та планування розвитку.

#### **1.2. Огляд системи `fintom8`**

`fintom8` — це високопродуктивний асинхронний мікросервіс, розроблений на фреймворку FastAPI. Його основне завдання — надійна та швидка обробка документів у форматах PDF та XML через REST API для інтеграції з іншими сервісами та додатками.

#### **1.3. Бізнес-контекст**

У цифровому документообігу компанії стикаються з необхідністю обробки великих обсягів структурованих (XML) та неструктурованих (PDF) документів. Ручна обробка є повільною, дорогою та схильною до помилок.

`fintom8` автоматизує ключові процеси обробки документів:
*   **Стандартизація:** Перетворення вхідних PDF-документів у формат PDF/A, що є вимогою для довготривалого архівного зберігання.
*   **Валідація:** Перевірка відповідності XML-документів (наприклад, електронних рахунків-фактур) затвердженим XSD-схемам.
*   **Вилучення даних:** Автоматичне вилучення текстового вмісту з PDF для подальшого аналізу чи індексації.

#### **1.4. Ключові можливості**

*   **Обробка PDF:**
    *   Конвертація до формату PDF/A (PDF/A-1b, PDF/A-2b).
    *   Вилучення текстового шару з документів.
*   **Обробка XML:**
    *   Валідація XML-файлів за наданою XSD-схемою.
    *   Вилучення даних з XML за допомогою XPath-запитів.
*   **Асинхронна обробка:** Виконання довготривалих операцій (конвертація великих файлів) у фоновому режимі без блокування API.
*   **Надійність:** Система розроблена з урахуванням обробки помилок та забезпечення стабільної роботи.

#### **1.5. Термінологія**

*   **PDF/A (PDF for Archiving):** Стандартизований ISO-формат PDF, призначений для довготривалого архівування електронних документів.
*   **XSD (XML Schema Definition):** Мова для опису структури та обмежень вмісту XML-документа.
*   **FastAPI:** Сучасний веб-фреймворк для створення API на Python.
*   **Endpoint:** Кінцева точка API, доступна за унікальним URL, яка виконує певну операцію.
*   **Celery:** Розподілена черга завдань для виконання асинхронних операцій.
*   **Celery Worker:** Фоновий процес, що виконує завдання з черги Celery.

### **2. Архітектура системи**

#### **2.1. Загальна архітектурна діаграма**

`[Діаграма C4: Контекст системи]`

**Опис діаграми:**
Система `fintom8` взаємодіє з такими зовнішніми компонентами:
1.  **Клієнтський додаток:** Веб-інтерфейс або інший бекенд-сервіс, що використовує REST API `fintom8`.
2.  **Сховище файлів (S3/MinIO):** Об'єктне сховище для оригінальних та оброблених документів.
3.  **База даних (PostgreSQL):** Зберігає метадані про документи, статуси завдань та API-ключі.
4.  **Брокер повідомлень (Redis):** Використовується Celery для організації черги асинхронних завдань.

#### **2.2. Архітектура компонентів**

`[Діаграма C4: Контейнери або діаграма компонентів]`

**Опис компонентів:**
Сервіс `fintom8` має багатошарову архітектуру для чіткого розподілу відповідальності.

*   **2.2.1. API Layer (FastAPI)**
    *   **Відповідальність:** Приймає HTTP-запити, валідує їх за допомогою Pydantic, проводить автентифікацію та серіалізує відповіді. Є єдиною точкою входу до системи.
*   **2.2.2. Service Layer (Processing Core)**
    *   **Відповідальність:** Оркеструє бізнес-логіку. Визначає, який модуль обробки (PDF чи XML) викликати. Для довготривалих операцій створює асинхронне завдання та передає його до черги Celery.
*   **2.2.3. PDF Processing Module (`pdf_processor`)**
    *   **Відповідальність:** Інкапсулює логіку, специфічну для PDF. Взаємодіє з `PyMuPDF` для вилучення тексту та викликає зовнішній процес `Ghostscript` для конвертації в PDF/A.
*   **2.2.4. XML Processing Module (`xml_processor`)**
    *   **Відповідальність:** Реалізує логіку валідації XML-файлів за XSD-схемами з використанням бібліотеки `lxml`.
*   **2.2.5. Async Task Workers (Celery)**
    *   **Відповідальність:** Виконують ресурсомісткі та довготривалі завдання (напр., конвертація PDF) у фоні, не блокуючи основний API-сервіс. Після завершення оновлюють статус завдання в базі даних.
*   **2.2.6. Data Access Layer (DAL)**
    *   **Відповідальність:** Абстрагує взаємодію з базою даних за допомогою SQLAlchemy Core. Надає методи для створення, читання та оновлення записів.

#### **2.3. Потоки даних**

**2.3.1. Потік обробки PDF (асинхронний):**
1.  Клієнт надсилає `POST` запит на `/documents/pdf/convert-to-pdfa` з файлом.
2.  **API Layer** приймає запит та валідує API-ключ.
3.  **Service Layer** зберігає оригінальний файл у S3, створює записи в таблицях `documents` та `processing_jobs` зі статусом `pending`.
4.  **Service Layer** надсилає завдання `convert_pdf_to_pdfa` в чергу Celery з `job_id`.
5.  **API Layer** негайно повертає клієнту відповідь `202 Accepted` з `job_id`.
6.  **Celery Worker** отримує завдання з черги.
7.  **Worker** завантажує файл з S3.
8.  **Worker** викликає `pdf_processor` для конвертації.
9.  **Worker** завантажує результат конвертації в S3.
10. **Worker** оновлює запис у `processing_jobs`: статус змінюється на `success` (з посиланням на результат) або `failed` (з описом помилки).

**2.3.2. Потік валідації XML (синхронний):**
1.  Клієнт надсилає `POST` запит на `/documents/xml/validate` з XML-файлом та ID схеми.
2.  **API Layer** приймає та валідує запит.
3.  **Service Layer** викликає `xml_processor`.
4.  `xml_processor` завантажує відповідну XSD-схему та виконує валідацію файлу.
5.  **API Layer** повертає клієнту відповідь `200 OK` з результатом валідації або `422 Unprocessable Entity` з деталями помилок.

### **3. Технічний стек**

*   **Мова програмування:** Python 3.10+
*   **Фреймворк:** FastAPI (на основі Starlette та Pydantic), обраний за високу продуктивність, вбудовану асинхронність та автоматичну генерацію OpenAPI-документації.
*   **Ключові бібліотеки:**
    *   **Веб-сервер:** Uvicorn (ASGI).
    *   **Обробка PDF:** `PyMuPDF (fitz)` для вилучення тексту, `Ghostscript` (зовнішня залежність) для конвертації в PDF/A.
    *   **Обробка XML:** `lxml` для парсингу та валідації.
    *   **Асинхронні задачі:** Celery з Redis як брокером повідомлень.
*   **База даних:**
    *   **Основна:** PostgreSQL 14+, обраний за надійність та зрілу екосистему.
    *   **Драйвер та міграції:** SQLAlchemy Core для гнучкості, Alembic для міграцій схеми.
*   **Сховище файлів:** Amazon S3, MinIO або сумісні об'єктні сховища.
*   **Інструменти розробки:**
    *   **Контейнеризація:** Docker, Docker Compose.
    *   **Менеджер залежностей:** Poetry.

### **4. Функціональність**

#### **4.1. Модуль обробки PDF**

*   **4.1.1. Конвертація в PDF/A**
    *   **Опис:** Сервіс конвертує PDF-файли у формат, сумісний зі стандартами PDF/A-1b та PDF/A-2b.
    *   **Реалізація:** Процес виконується за допомогою зовнішньої утиліти `Ghostscript`, яка має бути встановлена в Docker-образі Celery-воркера.
    *   **Приклад виклику Ghostscript:**
        ```bash
        gs -dPDFA=2 -dBATCH -dNOPAUSE -sProcessColorModel=DeviceCMYK \
           -sDEVICE=pdfwrite -sOutputFile=output_pdfa.pdf input.pdf
        ```
    *   **Обмеження:** Якість конвертації залежить від структури вхідного PDF. Документи зі складними шрифтами або прозорістю можуть оброблятися некоректно.

*   **4.1.2. Вилучення тексту**
    *   **Опис:** Вилучення текстового вмісту, що зберігається у текстовому шарі PDF-документа.
    *   **Реалізація:** Використовується бібліотека `PyMuPDF`. Метод `page.get_text()` ітерується по всіх сторінках та об'єднує текст.
    *   **Примітка:** Оптичне розпізнавання символів (OCR) для зображень без текстового шару не підтримується.

#### **4.2. Модуль обробки XML**

*   **4.2.1. Валідація за XSD-схемою**
    *   **Опис:** Перевірка відповідності структури та вмісту XML-файлу правилам, визначеним у XSD-схемі.
    *   **Реалізація:** Використовується `lxml.etree`. XSD-схеми зберігаються на файловій системі сервісу та кешуються в пам'яті для пришвидшення наступних валідацій.
        ```python
        # Приклад логіки валідації
        from lxml import etree

        def validate_xml(xml_path: str, xsd_path: str) -> tuple[bool, list]:
            xml_doc = etree.parse(xml_path)
            xsd_doc = etree.parse(xsd_path)
            xmlschema = etree.XMLSchema(xsd_doc)
            try:
                xmlschema.assertValid(xml_doc)
                return True, []
            except etree.DocumentInvalid as e:
                return False, list(e.error_log)
        ```

#### **4.3. Приклади використання (Use Cases)**

*   **UC-01:** Бухгалтерська система завантажує XML-рахунок для валідації перед відправкою до податкової служби.
*   **UC-02:** Система документообігу завантажує PDF-контракт для конвертації у PDF/A перед переміщенням до електронного архіву.
*   **UC-03:** Аналітичний сервіс вилучає текстовий вміст із PDF-звіту для подальшого аналізу ключових слів.

### **5. API та інтеграції**

#### **5.1. Загальна специфікація API**

*   **Базовий URL:** `https://api.fintom8.com/v1/`
*   **Автентифікація:** За допомогою API-ключа, що передається в HTTP-заголовку `X-API-KEY`.
    ```
    X-API-KEY: your_secret_api_key_here
    ```
*   **Формати даних:** `application/json` для відповідей, `multipart/form-data` для завантаження файлів.
*   **Коди стану HTTP:**
    *   `200 OK`: Успішний синхронний запит.
    *   `202 Accepted`: Асинхронне завдання успішно прийнято в обробку.
    *   `400 Bad Request`: Некоректний запит (напр., відсутній файл).
    *   `401 Unauthorized`: Недійсний або відсутній API-ключ.
    *   `422 Unprocessable Entity`: Помилка валідації даних (напр., XML не відповідає схемі).
    *   `500 Internal Server Error`: Внутрішня помилка сервера.

#### **5.2. Опис кінцевих точок (Endpoints)**

*Інтерактивна OpenAPI-документація доступна за ендпоінтом `/docs`.*

*   **5.2.1. `POST /documents/pdf/convert-to-pdfa`**
    *   **Опис:** Запускає асинхронну задачу конвертації PDF у PDF/A.
    *   **Тіло запиту:** `multipart/form-data` з полем `file` (PDF-документ).
    *   **Приклад запиту (`curl`):**
        ```bash
        curl -X POST "https://api.fintom8.com/v1/documents/pdf/convert-to-pdfa" \
             -H "X-API-KEY: your_secret_api_key_here" \
             -F "file=@/path/to/document.pdf"
        ```
    *   **Приклад відповіді (`202 Accepted`):**
        ```json
        {
          "job_id": "a1b2c3d4-e5f6-7890-1234-567890abcdef"
        }
        ```

*   **5.2.2. `POST /documents/xml/validate`**
    *   **Опис:** Валідує XML-файл за вказаною схемою.
    *   **Тіло запиту:** `multipart/form-data` з полями `file` (XML-файл) та `schema_id` (ідентифікатор XSD-схеми).
    *   **Приклад запиту (`curl`):**
        ```bash
        curl -X POST "https://api.fintom8.com/v1/documents/xml/validate" \
             -H "X-API-KEY: your_secret_api_key_here" \
             -F "file=@/path/to/invoice.xml" \
             -F "schema_id=invoice_v1"
        ```
    *   **Приклад відповіді (`200 OK`):**
        ```json
        {
          "valid": true,
          "errors": []
        }
        ```

*   **5.2.3. `GET /jobs/{job_id}`**
    *   **Опис:** Отримує статус асинхронної задачі.
    *   **Приклад запиту (`curl`):**
        ```bash
        curl -X GET "https://api.fintom8.com/v1/jobs/a1b2c3d4-e5f6-7890-1234-567890abcdef" \
             -H "X-API-KEY: your_secret_api_key_here"
        ```
    *   **Приклади відповідей (`200 OK`):**
        *   Завдання в процесі: `{"job_id": "...", "status": "pending"}`
        *   Завдання успішно виконано:
            ```json
            {
              "job_id": "...",
              "status": "success",
              "result_url": "https://s3.fintom8.com/processed/document.pdfa.pdf"
            }
            ```
        *   Помилка виконання:
            ```json
            {
              "job_id": "...",
              "status": "failed",
              "error_message": "Failed to convert document: invalid format."
            }
            ```

### **6. База даних**

#### **6.1. Схема даних**

`[ER-діаграма]`

**Опис зв'язків:**
*   `api_keys` (1) ---< `documents` (N): Один API-ключ може бути пов'язаний з багатьма документами.
*   `documents` (1) ---< `processing_jobs` (N): Один документ може мати кілька завдань обробки.

#### **6.2. Опис таблиць**

*   **`processing_jobs`**: Зберігає інформацію про асинхронні завдання обробки.
    *   `id` (UUID, PK): Унікальний ідентифікатор завдання.
    *   `document_id` (UUID, FK): Посилання на `documents`.
    *   `type` (VARCHAR): Тип завдання (напр., `pdf_to_pdfa`, `extract_text`).
    *   `status` (VARCHAR): Статус (`pending`, `in_progress`, `success`, `failed`).
    *   `result_url` (VARCHAR, nullable): URL до обробленого файлу.
    *   `error_message` (TEXT, nullable): Опис помилки.
    *   `created_at`, `updated_at` (TIMESTAMP).

*   **`documents`**: Зберігає метадані про завантажені документи.
    *   `id` (UUID, PK): Унікальний ідентифікатор документа.
    *   `original_filename` (VARCHAR): Оригінальне ім'я файлу.
    *   `storage_path` (VARCHAR): Шлях до файлу в S3.
    *   `mime_type` (VARCHAR): MIME-тип файлу.
    *   `api_key_id` (UUID, FK): Посилання на `api_keys`.

*   **`api_keys`**: Зберігає хеші API-ключів для автентифікації.
    *   `id` (UUID, PK): Унікальний ідентифікатор ключа.
    *   `key_hash` (VARCHAR): Хеш API-ключа (SHA-256).
    *   `user_id` (VARCHAR): Ідентифікатор користувача/системи.
    *   `is_active` (BOOLEAN): Прапорець активності.
    *   `description` (TEXT, nullable): Опис призначення ключа.

#### **6.3. Процедура міграції схеми**

Для управління змінами в схемі бази даних використовується **Alembic**.

*   Створення нової міграції:
    ```bash
    poetry run alembic revision --autogenerate -m "Add description to api_keys"
    ```
*   Застосування міграцій:
    ```bash
    poetry run alembic upgrade head
    ```

### **7. Безпека**

#### **7.1. Автентифікація та авторизація**
Всі запити до API вимагають валідного API-ключа, переданого в заголовку `X-API-KEY`. Ключі в базі даних зберігаються у вигляді хешів (SHA-256), що унеможливлює їх відновлення.

#### **7.2. Захист даних**
*   **При передачі:** Весь трафік шифрується за допомогою TLS 1.2+.
*   **При зберіганні:** Файли в S3/MinIO захищені за допомогою шифрування на стороні сервера (Server-Side Encryption).

#### **7.3. Безпека API**
*   **Валідація вхідних даних:** FastAPI та Pydantic автоматично валідують усі вхідні дані, запобігаючи атакам типу ін'єкцій.
*   **Обмеження розміру файлу:** На рівні веб-сервера (Nginx) встановлено обмеження на максимальний розмір файлу, що завантажується.
*   **Обмеження швидкості запитів (Rate Limiting):** Реалізовано для запобігання DoS-атакам.

#### **7.4. Управління секретами**
Секрети (ключі доступу, паролі) заборонено зберігати у кодовій базі.
*   **Локальна розробка:** Використовується файл `.env`.
*   **Продакшн:** Секрети передаються через змінні середовища, які управляються системами оркестрації (Kubernetes Secrets) або спеціалізованими сервісами (HashiCorp Vault, AWS Secrets Manager).

### **8. Розгортання та інфраструктура**

#### **8.1. Середовища**
*   `development`: Локальне середовище розробника (Docker Compose).
*   `staging`: Середовище, ідентичне продакшну, для фінального тестування.
*   `production`: Робоче середовище для кінцевих користувачів.

#### **8.2. Процес розгортання**
Додаток розгортається у вигляді Docker-контейнерів, оркестрованих за допомогою Kubernetes або Docker Swarm.
Існує два типи контейнерів:
1.  **`app`:** FastAPI-додаток для обслуговування API-запитів.
2.  **`worker`:** Celery-воркер для обробки фонових завдань.

#### **8.3. CI/CD Pipeline**
Пайплайн (GitHub Actions / GitLab CI) складається з таких етапів:
1.  **Lint & Format:** Перевірка коду за допомогою `black` та `flake8`.
2.  **Unit Tests:** Запуск unit-тестів за допомогою `pytest`.
3.  **Build:** Збірка Docker-образів `app` та `worker`.
4.  **Push:** Завантаження образів у Docker Registry (ECR, Docker Hub).
5.  **Deploy to Staging:** Автоматичне розгортання в `staging`-середовищі.
6.  **Integration Tests:** Запуск інтеграційних тестів на `staging`.
7.  **Deploy to Production:** Розгортання в `production`-середовищі (ручне або автоматичне після затвердження).

### **9. Тестування**

#### **9.1. Стратегія тестування**
Застосовується підхід "піраміди тестування" з акцентом на unit-тестах, доповнених інтеграційними та E2E-тестами.

#### **9.2. Типи тестів**
*   **Unit-тести (`pytest`):** Тестують окремі функції та класи в ізоляції.
*   **Інтеграційні тести (`TestClient`):** Тестують взаємодію компонентів системи (напр., запит до API → створення завдання в БД → відправка в чергу).
*   **End-to-End тести:** Тестують повний цикл обробки документа, від API-запиту до перевірки результату в S3.

#### **9.3. Покриття коду**
Використовується `pytest-cov` для вимірювання покриття коду тестами. Цільовий показник — не менше 85%.

### **10. Моніторинг та логування**

#### **10.1. Логування**
*   **Формат:** Усі логи пишуться у форматі JSON для спрощення парсингу.
*   **Агрегація:** Логи з усіх контейнерів централізовано збираються в ELK Stack (Elasticsearch, Logstash, Kibana) або Grafana Loki.

#### **10.2. Моніторинг**
*   **Ключові метрики:**
    *   Кількість запитів на секунду (RPS) та час відповіді API (latency).
    *   Відсоток HTTP-помилок (4xx, 5xx).
    *   Довжина черги Celery, кількість успішних та невдалих завдань.
    *   Споживання CPU/RAM контейнерами.
*   **Інструменти:** Prometheus для збору метрик, Grafana для візуалізації.

#### **10.3. Сповіщення**
Налаштовані сповіщення (Alertmanager), що відправляють повідомлення в Slack/PagerDuty у разі критичних подій:
*   Високий відсоток 5xx-помилок.
*   Значне зростання часу відповіді API.
*   Переповнення черги Celery.
*   Недоступність бази даних або Redis.

### **11. Посібник для розробників**

#### **11.1. Початок роботи**

1.  **Передумови:** Python 3.10+, Poetry, Docker, Docker Compose.

2.  **Клонування репозиторію:**
    ```bash
    git clone https://github.com/your-org/fintom8.git
    cd fintom8
    ```

3.  **Встановлення залежностей:**
    ```bash
    poetry install
    ```

4.  **Налаштування середовища:**
    Скопіюйте `.env.example` у `.env` та заповніть необхідні значення.
    ```bash
    cp .env.example .env
    ```
    Приклад `.env`:
    ```dotenv
    DATABASE_URL=postgresql://user:password@localhost:5432/fintom8_db
    REDIS_URL=redis://localhost:6379/0
    SECRET_KEY=your_very_secret_key
    ```

5.  **Запуск сервісу:**
    Використовуйте Docker Compose для запуску повного стеку.
    ```bash
    docker-compose up --build
    ```
    API буде доступний за адресою `http://localhost:8000`.

6.  **Запуск тестів:**
    ```bash
    poetry run pytest
    ```

#### **11.2. Процес розробки**
*   **Стиль коду:** Дотримуйтесь стандартів, визначених у `black`, `isort` та `flake8`. Налаштований pre-commit hook для автоматизації перевірок.
*   **Git Flow:** Нові функції розробляються у гілках `feature/*`, виправлення — у `hotfix/*`. Pull Request повинен містити опис змін та посилання на відповідне завдання.

#### **11.3. Вирішення поширених проблем (Troubleshooting)**
*   **Проблема:** `Ghostscript not found`.
    *   **Рішення:** Переконайтеся, що Ghostscript встановлений у Docker-образі `worker` (напр., `apt-get install ghostscript` у Dockerfile).
*   **Проблема:** Celery-воркер не отримує завдань.
    *   **Рішення:** Перевірте налаштування `REDIS_URL`. Переконайтеся, що Redis-контейнер запущений і доступний для воркера.
*   **Проблема:** Помилки міграції Alembic.
    *   **Рішення:** Перевірте з'єднання з базою даних. Переконайтеся, що локальна схема не розійшлася з історією міграцій. Для локальної розробки може знадобитися перестворення бази даних.
