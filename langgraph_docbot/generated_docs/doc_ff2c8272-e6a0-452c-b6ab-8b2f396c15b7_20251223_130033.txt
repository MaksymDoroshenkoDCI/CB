## 1. Introduction and Overview

This document provides a comprehensive technical overview of the Delivery-Car system. It details the system architecture, technology stack, features, and operational procedures required for development, deployment, and maintenance.

### 1.1 System Purpose and Main Goals

**Delivery-Car** is a web-based platform designed to streamline the process of purchasing vehicles from major US auctions, specifically **Copart** and **AIIA (Insurance Auto Auctions)**.

The primary goal of the project is to create a user-friendly, transparent, and efficient ecosystem for buying auction cars. It aims to solve the complexity and accessibility barriers often associated with auto auctions, making the process available to individuals without prior experience.

The key objectives are:
*   **Aggregation:** To provide a single interface for browsing and searching vehicle listings from multiple auction sources.
*   **Simplification:** To abstract the complexities of bidding, purchasing, and logistics into a simple, guided workflow.
*   **Accessibility:** To empower both individual buyers and professional dealers with powerful tools for finding and acquiring vehicles.

### 1.2 Target Audience

The system is designed to serve a diverse user base with distinct roles and needs:

*   **Private Buyers:** Individuals looking to purchase a vehicle for personal use. They require a simple interface, clear information, and guidance throughout the purchasing process.
*   **Car Dealers & Brokers:** Professionals who buy vehicles in larger volumes. They need advanced search filters, efficient order management, and detailed reporting functionalities.
*   **System Administrators:** Internal staff responsible for managing the platform, overseeing orders, monitoring system health, and managing integrations with auction houses.

### 1.3 Key Features Summary

The platform is built around the following core functionalities:

1.  **User Account Management:** Secure registration, login, and a personal dashboard ("cabinet").
2.  **Aggregated Car Search:** A powerful search engine to find vehicles from Copart and AIIA with various filters.
3.  **Detailed Vehicle View:** Comprehensive pages for each vehicle, displaying photos, specifications, auction details, and history reports.
4.  **Favorites List:** Ability for users to save interesting vehicles to a personal "Favorites" list for later review.
5.  **Order Placement & Management:** A streamlined workflow for placing an order (bidding request) on a vehicle.
6.  **Order Status Tracking:** Real-time tracking of the order status, from bid placement to vehicle delivery.
7.  **Purchase History:** A complete history of all past and current orders for each user.
8.  **Internal Chat:** A direct messaging system for users to communicate with platform managers or support staff.
9.  **Automated Notifications:** System-generated email or in-app notifications about important status changes and updates.

## 2. System Architecture

### 2.1 Overall Architecture Description

The Delivery-Car platform will be implemented using a **Monolithic API Backend** with a **Decoupled Frontend**. This architecture provides a clear separation of concerns between the user interface and the business logic, allowing for independent development and scaling.

*   **Frontend Application:** A Single Page Application (SPA) built with React/Next.js that communicates with the backend via a RESTful API. It handles all user interface rendering and client-side state management.
*   **Backend API:** A Python-based API built with the FastAPI framework. It exposes REST endpoints to handle all business logic, data processing, user authentication, and communication with external services.
*   **Database:** A PostgreSQL relational database serves as the primary data store for user information, orders, vehicle data, and other application-critical information.
*   **External Services:** The system integrates with the official APIs of Copart and AIIA to fetch vehicle listings and auction data.

### 2.2 Main Components and Modules

The system is logically divided into the following modules:

*   **Authentication Module:** Handles user registration, login, password management, and JWT generation/validation.
*   **User Module:** Manages user profiles, roles (Buyer, Dealer, Admin), and personal data.
*   **Auction Integration Module:** Contains services responsible for communicating with Copart and AIIA APIs, fetching data, and normalizing it for our system.
*   **Catalog Module:** Manages the vehicle data, search functionality, filtering, and detailed vehicle information display.
*   **Ordering Module:** Encapsulates all logic related to placing orders, managing bids, and tracking the order lifecycle.
*   **Notification Module:** Responsible for sending automated emails, push notifications, or in-app alerts.
*   **Chat Module:** Powers the real-time messaging functionality between users and managers.

### 2.3 System Diagrams (Text Format)

#### Component Diagram

```
[User's Browser] <--> [React/Next.js Frontend (SPA)]
       ^
       | (HTTPS/REST API Calls)
       v
[Backend API (FastAPI on Docker)]
       |
       +-----> [PostgreSQL Database]
       |
       +-----> [Copart REST API]
       |
       +-----> [AIIA REST API]
```

### 2.4 Data Flow

1.  A user accesses the web application through their browser. The React/Next.js frontend is loaded.
2.  The user performs a search for a vehicle. The frontend sends an API request to the FastAPI backend (e.g., `GET /api/v1/cars?make=toyota`).
3.  The backend's **Auction Integration Module** makes parallel requests to the Copart and AIIA APIs to fetch relevant vehicle data.
4.  The backend normalizes the data from both sources into a unified format and returns it to the frontend.
5.  When a user places an order, the frontend sends a `POST` request to `/api/v1/orders` with the vehicle ID and user details (via JWT).
6.  The backend's **Ordering Module** validates the request, creates an order record in the PostgreSQL database with an initial status (e.g., "Pending Review"), and returns a confirmation to the user.
7.  An administrator reviews the order via their admin panel. Upon approval, the status is updated. This triggers the **Notification Module** to send an alert to the user.
8.  The entire order lifecycle is managed through status updates in the database, which are reflected in the user's dashboard.

## 4. Functionality

### 4.1 Detailed Feature Descriptions

#### 4.1.1 User Account (Personal Cabinet)
*   **Description:** Users can register, log in, and access a personal dashboard.
*   **Business Logic:** The dashboard will display a summary of active orders, favorite vehicles, recent messages, and purchase history. Users can also edit their profile information and change their password.
*   **Use Case:** A registered user logs in to check the status of their recent order and view messages from their manager.

#### 4.1.2 Car Search and Filtering
*   **Description:** A central feature allowing users to search for vehicles across both Copart and AIIA.
*   **Business Logic:** The search will support multiple filters like make, model, year, location, damage type, and auction date. The backend will query external APIs in real-time or from a periodically updated cache.
*   **Use Case:** A car dealer searches for "Toyota Camry" models from 2018-2020 with "front-end damage" located in Florida.

#### 4.1.3 Order Placement and Tracking
*   **Description:** A step-by-step process for a user to place a request to bid on or buy a specific vehicle.
*   **User Workflow:**
    1.  User finds a car and clicks "Place Order".
    2.  A form appears to confirm vehicle details and enter a maximum bid amount.
    3.  User submits the form.
    4.  An order is created in the system with status "New".
    5.  The user can track the order's progress through statuses like "Manager Assigned", "Bid Placed", "Won", "Lost", "Payment Pending", "Shipped".
*   **Business Logic:** The system logs every status change with a timestamp. Admins are notified of new orders.

#### 4.1.4 Internal Chat
*   **Description:** A real-time chat interface for users to communicate with assigned managers.
*   **Business Logic:** The chat will be linked to specific orders. When an admin is assigned to an order, a chat channel is created between the user and the admin. WebSocket technology is recommended for real-time communication.
*   **Use Case:** A user asks their manager a question about the shipping cost for a vehicle they won at auction.

#### 4.1.5 Automated Notifications
*   **Description:** The system automatically sends notifications to users about important events.
*   **Business Logic:** Triggers will be set up for key status changes. For example, when an order status changes to "Won", the user receives an email and an in-app notification.
*   **Channels:** Email (e.g., via SendGrid or AWS SES), In-app notifications.

## 5. API and Integrations

### 5.1 Internal REST API (High-Level)

The backend will expose a versioned REST API. All endpoints will be prefixed with `/api/v1/`.

**Example Endpoints:**
*   `POST /api/v1/auth/token` - Authenticate and receive a JWT.
*   `POST /api/v1/users/` - Register a new user.
*   `GET /api/v1/users/me` - Get current user's profile.
*   `GET /api/v1/cars/` - Search for cars with query parameters.
*   `GET /api/v1/cars/{car_id}` - Get details for a specific car.
*   `POST /api/v1/orders/` - Create a new order.
*   `GET /api/v1/orders/` - Get a list of the user's orders.
*   `GET /api/v1/orders/{order_id}` - Get details for a specific order.

### 5.2 External Service Integrations

*   **Copart API:**
    *   **Purpose:** To fetch vehicle listings, auction details, and high-resolution images.
    *   **Mechanism:** Integration via their official REST API. Requires an API key and adherence to rate limits.
    *   **Data Caching:** To improve performance and respect rate limits, data fetched from Copart should be cached for a short period (e.g., 15-30 minutes).

*   **AIIA (IAAI) API:**
    *   **Purpose:** Similar to Copart, for fetching vehicle and auction data.
    *   **Mechanism:** Integration via their official REST API.
    *   **Data Normalization:** A crucial step will be to create a unified data model within our system, as the data structures from Copart and AIIA will differ. An adapter pattern should be used to map their respective responses to our internal `Car` model.

### 5.3 Data Exchange Formats

All API communications (internal and external) will use **JSON** as the data exchange format.

## 6. Database Design

### 6.1 Database Schema (High-Level)

The PostgreSQL database will contain the following core tables:

*   **`users`**:
    *   `id` (PK, UUID)
    *   `email` (VARCHAR, UNIQUE, NOT NULL)
    *   `hashed_password` (VARCHAR, NOT NULL)
    *   `full_name` (VARCHAR)
    *   `role` (ENUM: 'buyer', 'dealer', 'admin')
    *   `created_at` (TIMESTAMP)
    *   `updated_at` (TIMESTAMP)

*   **`cars`** (Cached data from auctions):
    *   `id` (PK, UUID)
    *   `lot_number` (VARCHAR, UNIQUE)
    *   `source` (ENUM: 'copart', 'aiia')
    *   `make`, `model`, `year`
    *   `vin` (VARCHAR)
    *   `odometer` (INTEGER)
    *   `damage_type` (VARCHAR)
    *   `auction_date` (TIMESTAMP)
    *   `location` (VARCHAR)
    *   `details_json` (JSONB) - To store raw data from the source API.

*   **`orders`**:
    *   `id` (PK, UUID)
    *   `user_id` (FK to `users.id`)
    *   `car_lot_number` (VARCHAR)
    *   `status` (VARCHAR, e.g., 'new', 'won', 'lost', 'shipped')
    *   `max_bid` (DECIMAL)
    *   `final_price` (DECIMAL)
    *   `created_at` (TIMESTAMP)

*   **`favorites`** (Join table):
    *   `user_id` (FK to `users.id`)
    *   `car_lot_number` (VARCHAR)
    *   `created_at` (TIMESTAMP)

*   **`chat_messages`**:
    *   `id` (PK, BIGSERIAL)
    *   `order_id` (FK to `orders.id`)
    *   `sender_id` (FK to `users.id`)
    *   `message_text` (TEXT)
    *   `created_at` (TIMESTAMP)

### 6.2 Relationships

*   One `User` can have many `Orders`.
*   One `User` can have many `Favorites`.
*   One `Order` is associated with one `User`.
*   One `Order` can have many `Chat_Messages`.

## 7. Security

### 7.1 Authentication Mechanisms

*   Authentication will be handled using **JWT (JSON Web Tokens)** via the **OAuth2 Password Bearer flow**.
*   When a user logs in with their email and password, the server validates the credentials and returns an `access_token` and a `refresh_token`.
*   The `access_token` is short-lived (e.g., 15-60 minutes) and must be included in the `Authorization` header for all protected API requests.
*   The `refresh_token` is long-lived (e.g., 7-30 days) and is used to obtain a new `access_token` without requiring the user to log in again.

### 7.2 Authorization Rules

*   API endpoints will be protected based on user roles. FastAPI's dependency injection system will be used to enforce these rules.
*   **Buyer/Dealer:** Can access their own profile, orders, and favorites. They can create new orders.
*   **Dealer:** May have access to additional bulk-action endpoints in the future.
*   **Admin:** Can access all data, manage users, and update the status of any order.

### 7.3 Data Protection

*   **Password Hashing:** User passwords will never be stored in plaintext. They will be hashed using a strong, modern algorithm like **bcrypt** or **Argon2**.
*   **HTTPS/TLS:** All communication between the client and the server will be encrypted using HTTPS.
*   **API Keys:** Keys for external services (Copart, AIIA) will be stored securely as environment variables or using a secrets management service (e.g., HashiCorp Vault, AWS Secrets Manager). They will not be hardcoded in the source code.
*   **Data Validation:** All incoming data will be rigorously validated using Pydantic models in FastAPI to prevent injection attacks and ensure data integrity.

## 8. Deployment and Infrastructure

### 8.1 Deployment Process

The application will be containerized using **Docker**.

1.  **Backend:** A `Dockerfile` will be created for the FastAPI application. It will install Python dependencies and run the application using Gunicorn/Uvicorn.
2.  **Frontend:** A multi-stage `Dockerfile` will be used to first build the static assets of the React/Next.js application, and then serve them using a lightweight web server like Nginx.
3.  **Composition:** `docker-compose.yml` will be used to orchestrate the backend, frontend, and PostgreSQL database containers for local development and simple production environments.

### 8.2 Infrastructure Requirements

*   **VPS (Virtual Private Server):** A Linux-based server (e.g., on DigitalOcean, AWS EC2, Vultr) with Docker and Docker Compose installed.
*   **Kubernetes (K8s):** For more complex, scalable deployments, a managed K8s cluster (e.g., AWS EKS, GKE, DOKS) is recommended. This would involve creating K8s deployment manifests, services, and ingresses.
*   **Database:** A managed PostgreSQL service (e.g., AWS RDS, DigitalOcean Managed Databases) is recommended for production to handle backups, scaling, and maintenance.

### 8.3 CI/CD Pipeline

A simple CI/CD pipeline using **GitHub Actions** will be implemented:

1.  **Trigger:** On push to `main` or `develop` branch.
2.  **Jobs:**
    *   **Lint & Test:** Run linters (e.g., `flake8`, `eslint`) and execute unit/integration tests (`pytest`).
    *   **Build:** Build Docker images for the frontend and backend.
    *   **Push:** Tag the images and push them to a container registry (e.g., Docker Hub, GitHub Container Registry, AWS ECR).
    *   **Deploy:** SSH into the production server and run `docker-compose pull` and `docker-compose up -d` to deploy the new versions. For K8s, this step would use `kubectl apply`.

#### Example `docker-compose.yml`

```yaml
version: '3.8'

services:
  backend:
    image: my-registry/delivery-car-backend:latest
    env_file: .env
    ports:
      - "8000:8000"
    depends_on:
      - db

  frontend:
    image: my-registry/delivery-car-frontend:latest
    ports:
      - "3000:80"

  db:
    image: postgres:14-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data/
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=password
      - POSTGRES_DB=delivery_car_db

volumes:
  postgres_data:
```

## 9. Testing

### 9.1 Testing Strategy

A multi-layered testing strategy will be adopted to ensure code quality and system stability.

### 9.2 Test Types

*   **Unit Tests:**
    *   **Backend:** Using `pytest`, test individual functions and business logic in isolation. Mock database calls and external API requests.
    *   **Frontend:** Using `Jest` and `React Testing Library`, test individual React components.
*   **Integration Tests:**
    *   **Backend:** Test the API endpoints by making HTTP requests to a running application connected to a test database. This validates the interaction between different modules.
*   **End-to-End (E2E) Tests:**
    *   Using a framework like **Cypress** or **Playwright**, simulate real user workflows in a browser. For example, a test would automate logging in, searching for a car, and placing an order.

### 9.3 Testing Tools

*   **Backend:** `pytest`, `httpx` (for client requests).
*   **Frontend:** `Jest`, `React Testing Library`.
*   **E2E:** `Cypress`, `Playwright`.

## 10. Monitoring and Logging

### 10.1 Monitoring Approach

*   **Application Metrics:** The FastAPI backend will expose a `/metrics` endpoint using a library like `starlette-prometheus`. This will provide metrics on request latency, error rates, and throughput.
*   **Infrastructure Metrics:** Monitor CPU, memory, and disk usage of the host server or Kubernetes cluster.
*   **Tools:** **Prometheus** for metrics collection and **Grafana** for creating dashboards and alerts.

### 10.2 Logging Strategy

*   **Structured Logging:** All log entries should be in a structured format like JSON. The Python `structlog` library is recommended for the backend.
*   **Log Aggregation:** Logs from all services should be collected and sent to a central logging platform like the **ELK Stack (Elasticsearch, Logstash, Kibana)** or a cloud-based service like **Datadog** or **Grafana Loki**.
*   **Log Levels:** Use appropriate log levels (DEBUG, INFO, WARNING, ERROR, CRITICAL) to control log verbosity in different environments.

#### Example Logging Configuration (Conceptual)

```python
# In FastAPI, configure structlog middleware
# This will add request context (path, method, client IP) to every log entry
import structlog

structlog.configure(
    processors=[
        structlog.stdlib.filter_by_level,
        structlog.stdlib.add_logger_name,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
    # ... other configurations
)
```

### 10.3 Error Handling

*   The API will return standard HTTP status codes (e.g., `400` for bad requests, `404` for not found, `500` for server errors).
*   All unhandled exceptions will be caught by a global exception handler, logged with a full stack trace, and returned to the client as a generic `500` error to avoid leaking implementation details.

## 11. Developer Guide

### 11.1 Getting Started

1.  **Clone the Repository:**
    ```bash
    git clone <repository_url>
    cd delivery-car
    ```
2.  **Prerequisites:** Ensure you have Docker and Docker Compose installed on your local machine.
3.  **Configuration:** Copy the example environment file (`.env.example`) to `.env` and fill in the required values (database credentials, API keys, etc.).
4.  **Run the Application:**
    ```bash
    docker-compose up --build
    ```
5.  **Access Services:**
    *   Backend API: `http://localhost:8000/docs` (for Swagger UI)
    *   Frontend App: `http://localhost:3000`
    *   Database: Connect via `localhost:5432` (port may vary)

### 11.2 Development Setup (without Docker)

#### Backend (FastAPI)

```bash
# 1. Navigate to the backend directory
cd backend/

# 2. Create a virtual environment
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Run the development server
uvicorn app.main:app --reload
```

#### Frontend (React)

```bash
# 1. Navigate to the frontend directory
cd frontend/

# 2. Install dependencies
npm install

# 3. Run the development server
npm start
```
