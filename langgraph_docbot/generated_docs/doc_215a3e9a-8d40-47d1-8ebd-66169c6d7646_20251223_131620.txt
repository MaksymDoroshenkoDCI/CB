## 1. Introduction and Overview

This document provides a comprehensive technical overview of the AutoBidder system. It outlines the system architecture, technology stack, features, and operational guidelines for development, deployment, and maintenance.

### 1.1. System Purpose and Main Goals

AutoBidder is a web-based platform designed to simplify the process of purchasing vehicles from major US auto auctions, specifically **Copart** and **AIIA**.

The primary goal of the system is to provide a unified, user-friendly interface where users can:
- Browse and search for vehicles from multiple auction sources.
- Compare vehicle details, history, and pricing.
- Place bids and manage auction participation.
- Finalize purchases and track the order delivery status.

The system aims to solve the problem of complexity and inaccessibility in the car auction market, making it approachable for both experienced dealers and first-time private buyers.

### 1.2. Target Audience

The system is designed for three primary user roles:

-   **Private Buyers:** Individuals looking to purchase a single vehicle for personal use. Their primary needs are a simple search interface, clear vehicle information, and straightforward order placement and tracking.
-   **Car Dealers & Brokers:** Professionals who purchase vehicles in larger volumes. They require advanced functionality, efficient workflows, detailed reporting, and robust order management tools.
-   **Administrators:** Internal staff responsible for managing the platform. Their tasks include overseeing orders, verifying statuses, managing user accounts, and monitoring the health of system integrations.

### 1.3. Key Features Summary

The core functionality of the AutoBidder platform includes:

-   **User Account Management:** A personal dashboard for users to manage their profile, view history, and track activity.
-   **Aggregated Auction Search:** A powerful search engine to find vehicles across both Copart and AIIA platforms with advanced filtering.
-   **Detailed Vehicle Pages:** Comprehensive information for each vehicle, including photos, specifications, condition reports, and bidding history.
-   **Favorites/Watchlist:** Ability for users to save vehicles of interest for later viewing.
-   **Order Placement:** A streamlined workflow for placing an order and initiating the purchase process.
-   **Order Status Tracking:** Real-time updates on the status of an order, from auction win to final delivery.
-   **Purchase History:** A complete log of all past and current orders for each user.
-   **Internal Chat:** A direct communication channel between users and platform managers for support and inquiries.
-   **Automated Notifications:** System-generated alerts (e.g., via email or in-app) about significant status changes or required actions.

## 2. System Architecture

### 2.1. Overall Architecture Description

The AutoBidder system is designed using a modern, decoupled three-tier architecture, consisting of a frontend client, a backend API service, and a relational database. This separation of concerns allows for independent development, scaling, and maintenance of each component.

-   **Frontend (Client-Side):** A Single Page Application (SPA) built with React/Next.js that provides the user interface. It communicates with the backend exclusively through a RESTful API.
-   **Backend (Server-Side):** A Python-based REST API built with the FastAPI framework. It handles all business logic, data processing, user authentication, and integration with external auction services.
-   **Database:** A PostgreSQL database serves as the single source of truth for all application data, including users, vehicles, orders, and bids.

### 2.2. Main Components and Modules

The backend will be organized into several logical service modules:

-   **Authentication Service:** Manages user registration, login, and JWT-based session management.
-   **User Service:** Handles user profiles, roles, and permissions.
-   **Auction Aggregator Service:** A dedicated module responsible for periodically fetching, parsing, and storing vehicle data from the Copart and AIIA APIs. This can be implemented using background tasks (e.g., Celery).
-   **Order Management Service:** Controls the entire lifecycle of an order, from creation to completion.
-   **Notification Service:** Manages the sending of automated emails and in-app notifications.
-   **Chat Service:** Implements the real-time chat functionality, potentially using WebSockets.

### 2.4. Data Flow

1.  **User Interaction:** A user accesses the web application in their browser. The React/Next.js frontend renders the UI.
2.  **API Communication:** User actions (e.g., searching for a car, placing an order) trigger asynchronous API requests from the frontend to the FastAPI backend. All requests are authenticated using a JWT sent in the `Authorization` header.
3.  **Business Logic:** The backend receives the request, validates the data and permissions, and executes the necessary business logic.
4.  **External Integration:** For auction data, the `Auction Aggregator Service` makes scheduled requests to the Copart and AIIA REST APIs. The retrieved data is cleaned, standardized, and stored in the PostgreSQL database.
5.  **Database Operation:** The backend services interact with the PostgreSQL database to create, read, update, or delete records (e.g., creating a new user, updating an order status).
6.  **Response:** The backend sends a JSON response back to the frontend, which then updates the UI to reflect the new state.

## 4. Functionality

### 4.1. Detailed Feature Descriptions

#### 4.1.1. User Account (Personal Cabinet)
-   Users can register, log in, and log out.
-   The dashboard displays a summary of recent activity, active orders, and favorite vehicles.
-   Users can edit their profile information (name, contact details).
-   Role-based views: Dealers see additional tools for bulk operations, while Admins have access to a full management panel.

#### 4.1.2. Car Search and Filtering
-   A central search bar allows searching by keywords (make, model, VIN).
-   Advanced filters include: auction source (Copart/AIIA), year, mileage, vehicle type, damage type, location, and auction date.
-   Search results are paginated and can be sorted by various criteria (e.g., price, auction end time).

#### 4.1.3. Order Placement and Tracking
-   Users can initiate an order from a vehicle's detail page.
-   The order form collects necessary information and confirms the bid amount or purchase price.
-   Each order is assigned a unique ID and a status (e.g., `Pending`, `Auction Won`, `In Transit`, `Delivered`).
-   Users can view the real-time status and history of their order in their personal cabinet.

#### 4.1.4. Internal Chat
-   A chat widget allows a user to start a conversation with an assigned manager directly from the order page.
-   The chat supports text messages and potentially file attachments (e.g., for documents).
-   Managers can view and respond to all user inquiries from the admin panel.

### 4.2. Use Cases

-   **UC-01: User Registers and Searches for a Car**
    1.  A new user visits the site and completes the registration form.
    2.  They receive a confirmation email and activate their account.
    3.  The user logs in and uses the search filters to find a specific car model.
    4.  They click on a result to view its detailed page and add it to their favorites.

-   **UC-02: Dealer Places an Order**
    1.  An authenticated user with the "Dealer" role finds a suitable vehicle.
    2.  They click the "Place Order" button, fill in their maximum bid, and confirm the terms.
    3.  The system creates a new order with the status `Pending` and notifies an administrator.

-   **UC-03: Administrator Updates Order Status**
    1.  An administrator logs into the admin panel and views the list of pending orders.
    2.  After winning the vehicle at auction, the admin updates the order status to `Auction Won`.
    3.  The system automatically sends a notification to the user about the status change.

## 5. API and Integrations

### 5.1. Internal API Endpoints (AutoBidder API)

The backend will expose a RESTful API. All endpoints will be prefixed with `/api/v1/`.

**Authentication**
-   `POST /api/v1/auth/register`: Register a new user.
-   `POST /api/v1/auth/login`: Authenticate and receive a JWT.
-   `POST /api/v1/auth/refresh`: Obtain a new access token using a refresh token.

**Cars**
-   `GET /api/v1/cars`: Search and filter cars.
-   `GET /api/v1/cars/{car_id}`: Get details of a specific car.

**Orders**
-   `POST /api/v1/orders`: Create a new order.
-   `GET /api/v1/orders`: Get a list of the current user's orders.
-   `GET /api/v1/orders/{order_id}`: Get details of a specific order.
-   `PATCH /api/v1/admin/orders/{order_id}`: (Admin only) Update an order's status.

**Favorites**
-   `POST /api/v1/favorites`: Add a car to favorites.
-   `DELETE /api/v1/favorites/{car_id}`: Remove a car from favorites.

### 5.2. External Service Integrations

The system's core functionality relies on integrating with third-party auction APIs.

-   **Copart API:** Used to fetch vehicle listings, details, auction schedules, and potentially bidding information. Requires an API key and adherence to rate limits.
-   **AIIA (Insurance Auto Auctions) API:** Similar to the Copart API, used for sourcing vehicle data from AIIA.

An `Auction Aggregator Service` will be responsible for communicating with these APIs. It will run as a periodic background task to fetch new data and update existing records, preventing real-time API calls during user searches and reducing latency.

### 5.3. Data Exchange Formats

-   **Internal API:** All communication between the frontend and backend will use **JSON**.
-   **External APIs:** Data exchange with Copart and AIIA will be in **JSON** format, as provided by their REST APIs. The aggregator service will transform the data from each source into a standardized internal model before storing it.

## 6. Database Design

### 6.1. Database Schema

The PostgreSQL database will be structured around several core tables.

### 6.2. Data Models (Simplified)

**`users` table**
-   `id` (PK, UUID)
-   `email` (VARCHAR, UNIQUE)
-   `password_hash` (VARCHAR)
-   `full_name` (VARCHAR)
-   `role` (ENUM: 'private', 'dealer', 'admin')
-   `created_at` (TIMESTAMP)
-   `updated_at` (TIMESTAMP)

**`cars` table**
-   `id` (PK, UUID)
-   `vin` (VARCHAR, UNIQUE)
-   `make` (VARCHAR)
-   `model` (VARCHAR)
-   `year` (INTEGER)
-   `mileage` (INTEGER)
-   `details_json` (JSONB) - For storing all other vehicle-specific data.
-   `source_auction` (ENUM: 'copart', 'aiia')
-   `source_id` (VARCHAR) - The ID of the car on the source platform.
-   `created_at` (TIMESTAMP)

**`orders` table**
-   `id` (PK, UUID)
-   `user_id` (FK to `users.id`)
-   `car_id` (FK to `cars.id`)
-   `status` (ENUM: 'pending', 'won', 'lost', 'paid', 'in_transit', 'delivered', 'cancelled')
-   `max_bid_amount` (DECIMAL)
-   `final_price` (DECIMAL, nullable)
-   `created_at` (TIMESTAMP)
-   `updated_at` (TIMESTAMP)

**`favorites` table** (Many-to-Many relationship)
-   `user_id` (FK to `users.id`)
-   `car_id` (FK to `cars.id`)
-   `created_at` (TIMESTAMP)
-   Primary Key: (`user_id`, `car_id`)

### 6.3. Relationships

-   A `user` can have many `orders`. (One-to-Many)
-   A `user` can have many `favorites`. (Many-to-Many via `favorites` table)
-   An `order` belongs to one `user` and one `car`. (Many-to-One)

## 7. Security

### 7.1. Authentication Mechanisms

-   **JWT (JSON Web Tokens):** Authentication will be stateless.
    1.  A user submits their credentials (`email` and `password`) to the `/auth/login` endpoint.
    2.  The server verifies the credentials against the hashed password in the database.
    3.  Upon success, the server generates a short-lived **access token** and a long-lived **refresh token**.
    4.  The access token is sent with every subsequent API request in the `Authorization: Bearer <token>` header.
    5.  The refresh token is used to obtain a new access token when the old one expires.

### 7.2. Authorization Rules

-   **Role-Based Access Control (RBAC):** Access to certain endpoints and features will be restricted based on user roles (`private`, `dealer`, `admin`).
    -   Example: Only users with the `admin` role can access `/api/v1/admin/*` endpoints.
    -   This will be implemented in FastAPI using dependencies that verify the role claim within the JWT payload.

### 7.3. Data Protection

-   **Password Hashing:** User passwords will never be stored in plaintext. They will be hashed using a strong, salted algorithm like **bcrypt**.
-   **HTTPS/TLS:** All communication between the client and server will be encrypted using HTTPS to prevent man-in-the-middle attacks.
-   **Input Validation:** All incoming data will be rigorously validated using **Pydantic** models in FastAPI to prevent injection attacks (SQLi, XSS).

## 8. Deployment and Infrastructure

### 8.1. Deployment Process

The application will be deployed using Docker containers, ensuring consistency across development, staging, and production environments.

1.  **Build:** The CI/CD pipeline builds Docker images for the frontend and backend applications.
2.  **Push:** The built images are tagged and pushed to a container registry (e.g., Docker Hub, AWS ECR).
3.  **Deploy:** The new images are pulled onto the production server(s), and the containers are restarted to run the new version.

### 8.2. Infrastructure Requirements

-   **VPS or Kubernetes Cluster:** The application can be deployed on a virtual private server (for simplicity) or a Kubernetes cluster (for scalability and resilience).
-   **PostgreSQL Database:** A managed PostgreSQL instance (e.g., AWS RDS) or a self-hosted instance.
-   **Nginx:** As a reverse proxy to route traffic to the appropriate frontend or backend service and handle SSL termination.
-   **Object Storage:** An S3-compatible service for storing user-uploaded files, if needed.

### 8.3. CI/CD Pipeline

A simple CI/CD pipeline can be configured using GitHub Actions:

-   **On push to `main` branch:**
    1.  Run linters and automated tests.
    2.  If tests pass, build Docker images.
    3.  Push images to the container registry.
    4.  Trigger a deployment script on the production server via SSH.

## 9. Testing

### 9.1. Testing Strategy

A multi-layered testing approach will be adopted to ensure code quality and system reliability.

### 9.2. Test Types

1.  **Unit Tests:**
    -   **Backend:** Use `pytest` to test individual functions and business logic in isolation.

2.  **Integration Tests:**

    -   Use a framework like **Cypress** or **Playwright** to simulate real user workflows in a browser, from logging in to placing an order.

### 9.3. Testing Tools

-   **Backend:** `pytest`, `httpx` (for client requests).
-   **Frontend:** `Jest`, `React Testing Library`.
-   **E2E:** `Cypress`, `Playwright`.

## 10. Monitoring and Logging

### 10.1. Monitoring Approach

-   **Metrics Collection:** Use **Prometheus** to scrape key metrics from the backend API, such as request latency, error rates (HTTP 5xx), and endpoint usage.
-   **Visualization:** Use **Grafana** to create dashboards for visualizing the collected metrics and setting up alerts for abnormal behavior (e.g., a spike in server errors).
-   **Health Checks:** The backend will expose a `/health` endpoint that monitoring tools can check to verify application health.

### 10.2. Logging Strategy

-   **Structured Logging:** All services will produce structured logs in JSON format. This makes them easily searchable and parsable.
-   **Centralization:** Logs from all containers will be aggregated into a central logging platform like the **ELK Stack** (Elasticsearch, Logstash, Kibana) or a cloud service like **Sentry**.
-   **Log Levels:** Use standard log levels (DEBUG, INFO, WARNING, ERROR, CRITICAL) to control log verbosity.

### 10.3. Error Handling

-   Unhandled exceptions in the backend will be caught by a global middleware, logged with a full stack trace to Sentry, and a generic 500 error response will be returned to the client.
