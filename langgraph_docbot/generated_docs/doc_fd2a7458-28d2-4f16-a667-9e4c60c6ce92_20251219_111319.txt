### **1. Вступ та загальний опис системи**

#### **1.1. Призначення системи**

Сервіс **fintom8** — це високопродуктивний бекенд-сервіс, розроблений для автоматизації процесів прийому, валідації та обробки фінансових документів у форматах PDF та XML. Система вирішує ключову бізнес-проблему: автоматизацію рутинних операцій, мінімізацію людських помилок та прискорення фінансових документообігів, зокрема при обробці вхідних рахунків-фактур.

**Основні функції системи:**
*   **Прийом документів:** Надання захищеного REST API для завантаження файлів.
*   **Валідація XML:** Перевірка XML-документів на відповідність визначеним XSD-схемам та додатковим бізнес-правилам.
*   **Обробка PDF:** Вилучення текстового вмісту та аналіз метаданих PDF-файлів.
*   **Конвертація в PDF/A:** Перетворення вхідних PDF-документів у формат PDF/A-2b для забезпечення довгострокового архівного зберігання.

#### **1.2. Цільова аудиторія документації**
Цей документ призначений для технічних фахівців, які беруть участь у розробці, розгортанні, підтримці та інтеграції сервісу `fintom8`:
*   **Backend-розробники:** для розуміння архітектури, API та внутрішньої логіки.
*   **Frontend-розробники:** для інтеграції клієнтських додатків з API сервісу.
*   **Системні архітектори:** для аналізу позиції системи в IT-ландшафті компанії.
*   **DevOps-інженери:** для налаштування CI/CD, розгортання та моніторингу.
*   **QA-інженери:** для розробки тестових сценаріїв та автоматизованих тестів.
*   **Технічні менеджери проекту:** для огляду технічної реалізації.

#### **1.3. Контекст та межі системи**

`fintom8` є компонентом більшої екосистеми обробки документів. Він функціонує як сервіс, що приймає необроблені документи від зовнішніх систем і передає структуровані, валідовані дані або готові для архівування файли до систем-споживачів.

**Системи, з якими `fintom8` взаємодіє:**
*   **Джерела даних:** CRM, ERP, клієнтські веб-портали, що надсилають документи.
*   **Споживачі даних:** Системи довгострокового зберігання (архіви), ERP-системи для подальшої обробки, аналітичні платформи.

**Функціонал, що не входить до зони відповідальності системи:**
*   **Користувацький інтерфейс (UI):** `fintom8` є виключно API-сервісом.
*   **Автентифікація та управління користувачами:** Сервіс делегує ці функції зовнішнім Identity-провайдерам, використовуючи API-ключі або JWT-токени для авторизації запитів.
*   **Реалізація специфічної бізнес-логіки клієнтів:** Система надає універсальні інструменти обробки.

#### **1.4. Термінологія та глосарій**

*   **Електронний рахунок (E-invoice):** Структурований XML-документ, що відповідає стандарту (напр., UBL, Factur-X).
*   **PDF/A (PDF for Archiving):** Стандартизований ISO-формат PDF для довгострокового зберігання. Усі ресурси (шрифти, зображення) вбудовані у файл.
*   **XSD (XML Schema Definition):** Мова для опису структури та обмежень XML-документа, що використовується для його валідації.
*   **Завдання (Job):** Одиниця роботи в системі, що відповідає процесу обробки одного документа. Має унікальний ідентифікатор та статус (напр., `pending`, `success`, `failed`).

### **2. Архітектура системи**

#### **2.1. Загальний огляд архітектури**

Система `fintom8` побудована на основі **модульної сервіс-орієнтованої архітектури**. Внутрішня структура розгортається як єдиний застосунок, проте логічно розділена на компоненти з чіткою зоною відповідальності. Цей підхід спрощує розробку та тестування на початкових етапах, водночас забезпечуючи можливість майбутньої еволюції в мікросервісну архітектуру (наприклад, винесення сервісу конвертації PDF в окрему одиницю).

Ресурсоємні операції, такі як конвертація файлів та вилучення тексту, виконуються **асинхронно** за допомогою черги завдань. Це дозволяє уникнути блокування API та гарантує швидку відповідь клієнту.

#### **2.2. Компоненти системи (на основі C4 Model)**

##### **2.2.1. Контекст системи (System Context)**

`fintom8` функціонує як центральний сервіс обробки, взаємодіючи з такими елементами:
*   **Клієнтські системи (ERP, CRM):** Надсилають документи через REST API та отримують результати обробки.
*   **Зовнішнє сховище файлів (S3/MinIO):** Використовується для надійного зберігання оригінальних та оброблених файлів.
*   **База даних (PostgreSQL):** Зберігає метадані документів, статуси та результати виконання завдань.

##### **2.2.2. Контейнери (Containers)**

Система складається з таких контейнерів, що розгортаються незалежно:

1.  **Веб-застосунок (FastAPI Application):** Приймає HTTP-запити, виконує швидку валідацію, створює та відправляє завдання в чергу.
2.  **Воркер (Celery Worker):** Асинхронно обробляє завдання з черги, виконуючи тривалі операції (валідація XML, конвертація PDF).
3.  **База даних (PostgreSQL):** Реляційна СУБД для зберігання стану системи.
4.  **Брокер повідомлень (Redis):** Використовується як брокер для Celery та для потреб кешування.

##### **2.2.3. Компоненти застосунку (Components)**

Контейнер **FastAPI Application** має таку внутрішню структуру:

*   **Шар API (API Layer):** Обробляє HTTP-запити, проводить автентифікацію та валідує вхідні дані за допомогою Pydantic.
*   **Диспетчер завдань (Task Dispatcher):** Формує та відправляє завдання до черги Celery.
*   **Модуль валідації XML (XML Validation Module):** Інкапсулює логіку перевірки XML-файлів за XSD-схемами.
*   **Модуль обробки PDF (PDF Processing Module):** Відповідає за вилучення тексту та метаданих з PDF.
*   **Сервіс конвертації (Conversion Service):** Абстракція для взаємодії з зовнішніми інструментами (напр., Ghostscript) для конвертації у PDF/A.

#### **2.3. Потоки даних (Data Flows)**

**Сценарій асинхронної обробки документа:**
1.  **Клієнт** надсилає `POST` запит на ендпоінт `/api/v1/documents/upload` з файлом.
2.  **FastAPI App** отримує запит, зберігає файл у тимчасове сховище (S3) та створює запис у таблиці `processing_jobs` зі статусом `PENDING`.
3.  **API** миттєво повертає клієнту відповідь `202 Accepted` з унікальним `job_id`.
4.  Завдання на обробку з `job_id` відправляється в чергу **Redis**.
5.  **Celery Worker** отримує завдання з черги.
6.  Воркер завантажує файл з S3 для обробки.
7.  Виконується відповідна операція (наприклад, валідація XML або конвертація PDF).
8.  Результат операції та оброблений файл (за наявності) зберігаються в S3.
9.  Статус завдання в таблиці `processing_jobs` оновлюється на `SUCCESS` або `FAILED`.
10. **Клієнт** відстежує статус завдання, періодично надсилаючи запити на ендпоінт `GET /api/v1/jobs/{job_id}`.

### **4. Функціональність**

#### **4.1. Основні модулі**

##### **4.1.1. Модуль прийому документів (Ingestion API)**
*   Реалізує ендпоінт `POST /api/v1/documents/upload`.
*   Приймає запити `multipart/form-data` з файлом та опціональними метаданими.
*   Виконує базові перевірки: максимальний розмір файлу, дозволені MIME-типи (`application/pdf`, `application/xml`).
*   Ініціює асинхронну обробку та повертає `job_id`.

##### **4.1.2. Модуль валідації XML**
*   Перевіряє XML на синтаксичну коректність (well-formedness).
*   Валідує документ на відповідність вказаній XSD-схемі. Схеми зберігаються та версіонуються в межах сервісу.
*   Підтримує розширення для додаткових бізнес-перевірок (наприклад, валідація контрольних сум).

##### **4.1.3. Модуль обробки PDF**
*   **Вилучення тексту:** Використовує `pdfminer.six` для отримання текстового шару. Розпізнавання сканованих документів (OCR) на даному етапі не підтримується.
*   **Конвертація в PDF/A:** Ініціює виклик системної утиліти `Ghostscript` з параметрами для створення PDF/A-2b сумісного файлу.
    *   *Приклад виклику:*
    ```bash
    gs -dPDFA=2 -dBATCH -dNOPAUSE -sProcessColorModel=DeviceCMYK \
       -sDEVICE=pdfwrite -sPDFACompatibilityPolicy=1 \
       -sOutputFile=output_a2b.pdf input.pdf
    ```

##### **4.1.4. Модуль управління завданнями (Task Manager)**
*   Реалізований на базі `Celery` для ізольованого виконання завдань.
*   Реалізовано механізм повторних спроб (`retry`) з експоненційною затримкою для обробки тимчасових помилок (напр., недоступність мережевого сховища). Максимальна кількість спроб — 3.

#### **4.2. Сценарії використання (Use Cases)**

*   **UC-01: Успішна валідація XML-рахунку**
    1.  Клієнт завантажує коректний XML-файл.
    2.  Система приймає файл і повертає `job_id`.
    3.  Асинхронне завдання успішно валідує файл за XSD-схемою.
    4.  Статус завдання оновлюється на **`SUCCESS`**. Результат містить повідомлення про успішну валідацію.

*   **UC-02: Конвертація PDF-файлу в PDF/A**
    1.  Клієнт завантажує PDF-файл з параметром `{"action": "convert_to_pdfa"}`.
    2.  Система приймає файл, повертає `job_id`.
    3.  Асинхронне завдання викликає `Ghostscript` і успішно конвертує файл.
    4.  Новий PDF/A-файл зберігається у сховищі.
    5.  Статус завдання оновлюється на **`SUCCESS`**. Результат містить посилання на завантаження нового файлу.

*   **UC-03: Помилка валідації XML**
    1.  Клієнт завантажує XML-файл з невірною структурою.
    2.  Система приймає файл, повертає `job_id`.
    3.  Під час валідації `lxml` повертає помилку.
    4.  Статус завдання оновлюється на **`FAILED`**. Результат містить детальне повідомлення про помилку (напр., `Element 'totalAmount': This element is not expected. Expected is ( taxAmount )`).

### **5. API та інтеграції**

#### **5.1. Опис REST API**

API сервісу `fintom8` відповідає специфікації OpenAPI 3.0. Інтерактивна документація автоматично генерується і доступна за ендпоінтами:
*   `/docs`: Swagger UI
*   `/redoc`: ReDoc

**Загальні принципи:**
*   **Версіонування:** Через префікс URL (`/api/v1/`).
*   **Формат даних:** `application/json` для всіх запитів та відповідей.
*   **Автентифікація:** За допомогою API-ключа, що передається в заголовку `X-API-Key`.

#### **5.2. Ключові ендпоінти (Endpoints)**

*   `POST /api/v1/documents/upload`
    *   **Опис:** Завантажує документ для асинхронної обробки.
    *   **Тіло запиту:** `multipart/form-data`.
    *   **Успішна відповідь:** `202 Accepted`
        ```json
        {
          "job_id": "a1b2c3d4-e5f6-7890-1234-567890abcdef"
        }
        ```

*   `GET /api/v1/jobs/{job_id}`
    *   **Опис:** Повертає статус та результат завдання на обробку.
    *   **Успішна відповідь:** `200 OK`
        ```json
        {
          "job_id": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
          "status": "SUCCESS",
          "created_at": "2023-10-23T10:00:00Z",
          "finished_at": "2023-10-23T10:00:05Z",
          "result": {
            "message": "Validation successful",
            "output_file_url": "/api/v1/documents/doc_id_123/download"
          }
        }
        ```

#### **5.3. Моделі даних API (Pydantic Models)**

Приклад моделі відповіді для статусу завдання:
```python
from pydantic import BaseModel, UUID4
from typing import Optional, Dict, Any
from datetime import datetime

class JobStatusResponse(BaseModel):
    job_id: UUID4
    status: str  # e.g., "PENDING", "PROCESSING", "SUCCESS", "FAILED"
    created_at: datetime
    finished_at: Optional[datetime]
    result: Optional[Dict[str, Any]]

    class Config:
        orm_mode = True
```

### **8. Розгортання та інфраструктура (DevOps)**

#### **8.1. Інфраструктура**
Сервіс розгортається у контейнеризованому середовищі.
*   **Середовища:**
    *   `development`: Локальне оточення на базі `docker-compose`.
    *   `staging`: Повністю відтворює production для тестування.
    *   `production`: Продуктивне середовище з реплікацією та моніторингом.
*   **Оркестрація:** Kubernetes використовується для управління контейнерами в `staging` та `production` середовищах.

#### **8.2. Процес розгортання (CI/CD)**
Процес CI/CD автоматизовано за допомогою **GitHub Actions** і включає такі етапи:
1.  **Push до feature-гілки:** Запуск лінтерів (`flake8`, `black`) та unit-тестів (`pytest`).
2.  **Створення Pull Request до `main`:** Повторний запуск тестів, розрахунок покриття коду. Потрібне схвалення від іншого розробника.
3.  **Злиття з `main`:**
    *   Збирається новий Docker-образ з тегом `latest`.
    *   Образ завантажується в Docker Registry (напр., AWS ECR).
    *   Виконується автоматичне розгортання на `staging`-середовище.
4.  **Створення Git-тегу:**
    *   Збирається Docker-образ з тегом версії (напр., `v1.1.0`).
    *   Розгортання на `production` запускається вручну після успішного тестування на `staging`.

#### **8.3. Конфігурація**
*   **Змінні середовища:** Уся конфігурація (підключення до БД, ключі до S3) керується через змінні середовища, відповідно до методології 12-Factor App.
*   **Управління секретами:** У `production` секрети (паролі, API-ключі) зберігаються в AWS Secrets Manager та безпечно передаються в Kubernetes pod'и під час запуску.

### **11. Документація для розробників**

#### **11.1. Початок роботи (Getting Started)**

**Вимоги:**
*   Docker та Docker Compose
*   Git
*   Python 3.10+ (для локальних скриптів)

**Кроки для локального запуску:**
1.  **Клонувати репозиторій:**
    ```bash
    git clone git@github.com:your-org/fintom8.git
    cd fintom8
    ```
2.  **Налаштувати змінні середовища:**
    ```bash
    cp .env.example .env
    ```
    Відредагуйте файл `.env`, якщо необхідно змінити порти або інші налаштування.

3.  **Запустити проект:**
    ```bash
    docker-compose up --build
    ```
    Ця команда збере образи та запустить усі контейнери (API, воркер, БД, Redis).

4.  **Перевірити роботу:**
    *   API буде доступний за адресою `http://localhost:8000`.
    *   Документація Swagger UI: `http://localhost:8000/docs`.

#### **11.2. Керівництво по стилю коду (Code Style Guide)**
*   **PEP 8:** Дотримання стандарту PEP 8 є обов'язковим.
*   **Форматування:** Код автоматично форматується за допомогою `black`.
*   **ЛІнтинг:** Перевірка коду здійснюється за допомогою `flake8`.
*   **Імпорти:** Сортування імпортів виконується за допомогою `isort`.

Для автоматизації цих перевірок налаштований pre-commit hook. Для його встановлення виконайте:
```bash
pip install pre-commit
pre-commit install
```

#### **11.3. Процес розробки та внеску (Contribution Guide)**
1.  Створіть нову гілку від `main` з описовою назвою (напр., `feat/add-pdf-metadata-extraction` або `fix/xml-validation-bug`).
2.  Розробіть функціонал та обов'язково покрийте його unit-тестами.
3.  Переконайтесь, що всі тести та лінтери проходять успішно локально.
4.  Створіть Pull Request (PR) до гілки `main`.
5.  В описі PR детально поясніть зроблені зміни.
6.  Дочекайтеся Code Review та схвалення від щонайменше одного члена команди перед злиттям.
